There is following problems without Generics:-
--------------------------------------------
- Type Safety (Runtime Error Problems)
- Manual Casting
- No Compile-Time Error Detection 

Generics:-
=========
- Generics were introduced in Java to provide type safety, eliminate runtime errors caused by type casting, and enable reusable and flexible code.
					or
- Generics allow us to write one class, method, or interface that can work with different types of data (like String, Integer, Double) safely.
							


- Why It Is Used:
	- To make code:
		- Safer (no wrong data types)
		- Reusable
		- Easy to maintain

- Real-Life Examples:
a.  Water Bottle
	- One bottle can store water, juice, or milk — same bottle, different liquids.

b. USB Drive
	- Same USB can store photos, videos, or documents.

c. School Bag
	- One bag can carry books, lunch, or clothes.



But Why Generics Beyond Collections?
=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-
1. Reusable Storage Class
- Imagine you want to create:
	- String Box
	- Integer Box
	- Double Box
- Without Generics:
	- You must write separate classes.

2 Banking System
- Imagine:
- Bank stores:
	- Account Number → Integer
	- Customer Name → String
	- Balance → Double
- You need reusable containers.
- Without Generics
	- You create:
		- IntegerContainer
		- StringContainer
		- DoubleContainer

3. Utility Methods
- Why generics in methods?
- Because we want generic reusable logic.
- Problem Without Generic Method:- 
- Print method for each type:

	void printString(String s) {
    		System.out.println(s);
	}


	void printInteger(Integer i) {
    		System.out.println(i);
	}

	void printDouble(Double d) {
    		System.out.println(d);
	}

- Bad:	
	- Repeated logic
	- Many methods

- With Generic Method

	public static <T> void print(T value) {
		System.out.println(value);
	}

- Usage:
	- print("Java");
	- print(100);
	- print(5.5);

- Now:
	- One method
	- Works for all types

				or 
- Generic types allow you to define a class, interface, or method with placeholders (type parameters) for the data types they will work with.

Advantage of Generics:
----------------------

1. Type Safety

--Compiler prevents you from adding wrong types (e.g., you can't add a String to List<Integer>).

2. No More Downcasting / ClassCastException

--You get the exact type back without casting.

3. Code Reusability

--Write once, use with any type.

4. Compile-time Bug Detection

--Errors are caught early instead of at runtime.

5. Cleaner and More Readable Code

Bounded Types (Constraints on Type Parameters):-
=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=-=

--In Generic Classes, we can apply the constraint or bound the type parameter for a particular range by using "extends" keyword.
							or
-- Bounded Type allows us to restrict what type can be passed to generic class.


A. Unbounded type
-------------------------

class Test<T>
{
}

--It is unbounded type (without any constraint), we can pass any type of parameter.

Test<String> t=new Test<>();
Test<Integer> t=new Test<>();


B. Bounded type:
----------------------


class Test<T extends X>
{
}


--If X is a class type then we can pass either X type elements or sub class elements of X type as parameter types.

--------

Multiple Bounds (using &):-
=-=-=-=-=-=-=-=-=
Multiple Bounds means restricting a generic type to extend one class and implement one or more interfaces.
	
						or
Multiple Bounds allow a generic type to have more than one restriction.

Syntax:-
--------
class Test<T extends Number & Serializable>
{
}


--Here Test class is able to allow the elements which must be either same as Number or sub classes to the number and must be the implementations of Serializable interface.

Test<Integer> t=new Test<Integer>();--> Valid
Test<Float> t=new Test<Float>();--> Valid
Test<String> t=new Test<String>--> Invalid.(Because String type is not a subtype of Number, but its implements the Serializable interface)


***Note:  The "extends" keyword will not allow two class types at a time.

class Test<T extends Number & Thread>
{
}


Status: Invalid


But the "extends" keyword is able to allow more than one interface type.

class Test<T extends Runnable & Serializable>
{
}

Statis: valid.


--The "extends" is able to allow both class type and interface type , but, first we have to specify class type then we have to specify Interface type. i.e first class name and then interface name.




class Test<T extends Number & Runnable>
{
}


Status: Valid.

class Test<T extends Runnable & Number>
{
}

Status: Invalid.

Covariance type in Java:
========================

--As we know that to the super class reference variable we can store any of its subclass object.


Example:


String s = "Hello";

Object obj = s;


--Covariance allows a type to accept its subtypes while preserving the "is-a" relationship.

--In Java arrays are the covariance types, means we can store array of any object to the reference variable of Object[] array also.


Example:
--------

String[] fruits = {"Apple", "Mango", "Banana", "Pineapple"};

Object[] objArray = fruits; //allowed


Problem:

objArray[0] = new A(); // Heap Pollution, ArrayStoreException

Here, no error at compile time; the compiler is not able to detect it, but at runtime, we get an exception.



Invariant:-
---------
Means does not follow the parent-child relationship. In Java, Generics are Invariant by default.


Example:
---------
		List<String> li = new ArrayList<>(List.of("A","B","C","D"));
		List<Object> li1 = li; // Compile time error


--To make our Collection work as a covariance type and make it flexible, we need to make use of a wildcard.

--With the collections, we can use the following 3 types of wildcards:

1. Unbounded Wildcard 

2. Subtype or upper bound wildcards: Covariance (for reading/producing)

3. Supertype or Lower bound wildcards: Contravariance (for writing/consuming)


 

PECS Principle (Producer Extends, Consumer Super)

--Producer (you only get/read values) : Use ? extends T
--Consumer (you only put/write values):  Use ? super T




1. Unbounded wildcard:
---------------------

List<?>


--In Java Generics, a wildcard (?) represents an unknown type.

--Unbounded Wildcard (?) means:  "any type"

--Unbounded wildcard is used when you don’t care about the type, but still want type safety.

--There are two scenarios where using unbounded wildcard is useful.

1. When writing a method that relies on Object class functionality, If a method uses only methods defined in the Object class (like toString(), equals()), then the exact type is not important.

2. If a method uses operations that do not depend on the generic type (such as size(), clear(), isEmpty()), then <?> can be used.


List<?> means a list typed to an unknown type. This could be a List<A>, a List<B>, a List<String> etc.

Since  you do not know what type the List is typed to, you can only read from the collection, and you can only treat the objects read as being Object instances.

Difference between <T> and <?>:-
=-=-=-=-===-=-=-==-=-=-=-=-=-=-=

| Feature       	| `<T>`                   	| `<?>`                    |
| ------------- 	| ----------------------- 	| ------------------------ |
| Meaning       	| Specific type parameter 	| Unknown type             |
| Type known?   	| Yes (compiler knows T)  	| No (unknown type)        |
| Add elements? 	| Yes                   	| No (except null)         |
| Use case      	| When type matters       	| When type doesn't matter |


**Why do we need wildcards when we already have <T>?
- Wildcards provide flexibility to accept multiple generic types without defining a specific type parameter, especially when the actual type is not important.


2. Subtype or upper bound wildcards: (? extends T)
--------------------------------------------------

--Using this we can achieve the covariance type of feature in collections also.

syntax:

--means a type that is either T or a child of T, 


Example (Employee):
------------------

	for(Employee employee: employees){
		System.out.println(employee);

Type Erasure in Java:
=-=-=-=-=-=-=-=-=-=-=

--Type Erasure is a mechanism used by Java to implement Generics.

--Type Erasure means removing Generic type information at compile time.


Definition:
-----------

--Type Erasure is the process of removing generic type information at compile time so that generic code is converted into non-generic code for JVM execution.

Need for Type Erasure:
---------------------

--Java introduced generics in Java 5 without changing JVM

So:

--JVM does not understand generics
--Compiler converts Generic code to normal code
-- No Generic Information at Runtime:


This ensures:

--Backward compatibility with older Java versions


How Type Erasure Works:
---------------------

Case 1: Unbounded Type

class Box<T> {
    T value;
}


After compilation:

class Box {
    Object value;
}


Case 2: Bounded Type:

class Box<T extends Number> {
    T value;
}

After compilation:

class Box {
    Number value;
}



Example:

List<String> list = new ArrayList<>();
list.add("Hello");

After type erasure:

List list = new ArrayList();
list.add("Hello");



Key Effects of Type Erasure:
-------------------------


1. Method Overloading Restriction


void fun1(List<Integer> list) {
		
}
	
void fun1(List<String> list) {
		
} // Not allowed


2. Type Casting Inserted by Compiler

List<String> slist = List.of("Java", "Pyhton", "C++");

String s = slist.get(0);
	
Compiler internally adds:

String s = (String) list.get(0);


Important Note

--Generics provide compile-time type safety, not runtime safety.



Gift Box Analogy:

At compile time:  box has label (String, Integer)
At runtime: label removed:  only box remains


Limitations of Generics:-
=-=-==-=-=-=-=-=-=-=-=-=
1. We can not use Primitive Types 
	- Generics does not support primitive types
		Wrong:
			List<int> list = new ArrayList<>();
		Correct:
			List<Integer> list = new ArrayList<>();
	- Why:-
		Generics works only with objects, primitive (int, double, char) are not the objects.

- That's why we use Wrapper classes

	Primitive	Wrapper
	int		Integer
	double		Double
	char		Character


2. We can not create Object using Type Parameter.

- You can not object of T directly.

-Wrong:

class Test<T> {
    
    T obj = new T(); // ERROR
    
}


Why?

- After type Erasure Compiler don't know:-

	- What type of T actually is?
- That's why:- 

	- new T()
	- Is not possible.

3. We can not create Array of Generic Type. 
	- Generic array creation is not allowed.

Wrong:

T[] arr = new T[5]; // ERROR

or

List<String>[] arr = new List<String>[5]; // ERROR

Why:-
- Arrays checks type on runtime.
- Generics checks type on compile-time.
- This create Mis-match problem.

4. We can not use Type parameter with Static members.

 Wrong:

class Test<T> {
    
    static T obj; // ERROR
    
}

Why:-
- Static members are class level.
- But decides on Object level.
- That why create a conflict.

5. We can not check Generic Type with "instanceof" operator.

Wrong:

if(obj instanceof List<String>) { } // ERROR

Edge case:-
---------
if(obj instanceof List<?>) { } // Allowed

Why:-

	- On Runtime:
		- List<String>
	- After Type Erasure:-
		- List
- That is why we do not know actual type of Generic.

6. Method Overloading using Generic Type is not allowed:-

Wrong:

- void fun(List<Integer> list) {}

- void fun(List<String> list) {} // ERROR

Why:-

After Type Erasure :

- void fun(List list)

- void fun(List list)

Same signature → Compile error.

7. We can not use Generic type with Exceptions:-


- Wrong:
	- class MyException<T> extends Exception { } // ERROR

Why:-
- Exception handling happens runtime.
- Generics at compile-time.
